# YasumaroがChrome Web Storeに並んでからの1ヶ月 — v6.0からv6.5.23までの変化

Yasumaro が Chrome Web Store に初めて公開されたのは 2026-06-18、バージョン 6.0.0 でした。そこから今日の 6.5.23 まで、コミット数にして252、リリースにして約40回。1ヶ月弱でこれだけの数字が積み上がっているのを見ると、自分でも「そんなに触っていたのか」と少し驚きます。

久しぶりに使い方を思い出したい人、あるいはこれから使い始める人に向けて、何がどう変わったのかを整理しておきます。

## いそがしい人向けの結論

- **Obsidianなしでも使えるようになりました**。ローカルにMarkdownファイルを直接ダウンロードする機能が追加され、REST APIのセットアップが不要になりました
- **履歴の保存先がSQLite（OPFS + FTS5）に統一され**、全文検索やタグクラスタ表示が高速になりました
- **プライバシー周りが大きく強化**され、PIIマスキングのサンプル表示、同意バージョン管理、監査ログが入りました
- 内部的には `storage.ts` や `sqlite.ts` の巨大ファイルがモジュール分割され、保守しやすくなっています（利用には影響しません）

ここから、それぞれの変化がなぜ起きたのかを見ていきます。

## Obsidianが「必須」から「任意」になった

v6.0.0 の時点では、YasumaroはObsidian Local REST APIに接続することを前提にした設計でした。使うにはObsidianを立ち上げ、Local REST APIプラグインを入れ、証明書を扱い、APIキーをコピーしてくる必要がありました。実際、v6.3.5〜v6.4.1にかけて `docs/OBSIDIAN_SETUP_GUIDE.md` を新規作成したり、証明書エラーのトラブルシューティングを厚くしたりと、この導入の壁を下げる作業にかなりの回数を割いています。

転機になったのが v6.5.14 の「ローカル Markdown 書き出しの idle 一括化」と、その前提になった PBI #07 の実装です。ここで実現したのは、**Obsidian REST APIを一切使わずに、閲覧履歴を日次のMarkdownファイルとしてブラウザのダウンロードフォルダに保存する**という経路でした。

なぜこれが意味を持つかというと、Obsidianを使うかどうかの判断を「インストール前」から「インストール後、気が向いたとき」に先送りできるようになったからです。ローカルMarkdown書き出しはダッシュボードのチェックボックスをONにするだけで動き、Obsidianのセットアップは一切要求されません。あとから「やっぱりObsidianで管理したい」と思えば、同じ設定画面からObsidian連携を追加すればいい。両方を同時に有効化することもできます。

書き出しタイミングも、当初は「記録時に自動で書き出す」というON/OFFのチェックボックス1つでしたが、v6.5.18で4つのラジオボタンに置き換わりました。

| モード | 動作 |
|-------|------|
| 手動のみ | 自動書き出しをしない。エクスポートボタンを押したときだけ |
| 即時 | 記録直後、最短1分間隔で書き出す |
| アイドル時 / 30分ごと | ブラウザがアイドル状態になったとき、または最大30分ごとにまとめて書き出す |
| 日付が変わったとき | 日付が変わったタイミングで前日分をまとめて回収する |

「即時」だと記録するたびにダウンロード通知が出て煩わしい、という声を受けて、バッファに溜めてまとめて書き出す「アイドル時」「日付が変わったとき」を後から追加した形です。ブラウザのダウンロード完了通知が気になる場合は、`chrome://settings/downloads` から「ダウンロードが完了したとき、ダウンロード一覧を表示する」をOFFにすると静かになります。

## 履歴の保存先がSQLiteに一本化された

v6.0.0当時、閲覧履歴は `chrome.storage.local` に保存されていました。保持期間や最大件数の設定はこの時点であったものの、検索やタグ分析のような重い処理をするには不向きな構造です。

v6.3.0前後からOPFS（Origin Private File System）上でwa-sqliteを動かす基盤が整い始め、v6.5.9で診断メタデータ（トークン数・処理時間・AIプロバイダ名など）がSQLiteスキーマに正式に組み込まれました。これによって「SQLite History」パネルで、AIがどれだけのトークンを使ったか、処理にどれくらいかかったかまで一覧で確認できるようになっています。

移行時の設計判断として印象的だったのが v6.5.9 での修正です。当初、「記録履歴をSQLiteへ変換」ボタンは `chrome.storage.local` の元データを移行と同時に削除していました。これは移行が失敗したときに元データが失われるリスクがあったため、**移行はコピー操作に変更し、削除は診断パネルから明示的に実行する別ボタン**に分離しました。データを扱う機能では、統合と削除を同じボタンに乗せないほうが安全だという教訓です。

SQLite化の副産物として、Tag Cluster（タグの関連性をグラフ表示する機能）にパン・ズーム機能が付いたり、監査ログでAI要約の生成・保存操作を追跡できるようになったりしています。

## プライバシー機能がひとつずつ積み上がった

v6.0.0の時点でもプライバシー同意モーダルはありましたが、拒否した場合の扱いに問題がありました。v6.0.1で「3回拒否で永久に非表示」という挙動をGDPR第7条に沿うよう「30日後に再表示」に修正し、v6.3.0でプライバシーポリシー更新時の再同意フローが入りました。

v6.5.20では未同意状態がツールバーバッジで可視化されるようになり、プライバシー設定画面にPIIマスキングのサンプル表示が追加されました。「クラウドAIに送信する前に、実際どんなデータが匿名化されるのか」を固定サンプルで確認できるようになったのは、ブラックボックスになりがちなプライバシー処理を見える化する試みです。

## 地味だが効いている変更

- **リブランディング**: v6.0.2でソースコード・ドキュメント上の旧称「Obsidian Weave」を「Yasumaro」に一括修正
- **アクセシビリティ**: v6.3.8でaxe-coreによるWCAGチェックをCIに導入、v6.5.23で`inert`属性を使った非表示タブパネルの修正など、細かい修正が継続的に入っている
- **セキュリティ**: v6.5.20で `@subframe7536/sqlite-wasm` のバージョンをピン留めし、サプライチェーンリスクを低減。GitHub Actionsのアクションも全SHAピン留め済み

## これから使う人・戻ってくる人へ

v6.0時代を知っている人がいちばん実感するのは、「Obsidianの証明書設定で詰まった記憶」がもう不要になっているという点だと思います。ダッシュボードでローカルMarkdown書き出しをONにするだけで、その日の記録が `~/Downloads/Yasumaro/YYYY-MM-DD.md` に溜まっていきます。Obsidianへの本格連携は、気が向いたときに追加すればいいわけです。

初めて使う人向けの導入〜日常運用ガイドは別記事にまとめる予定です。
