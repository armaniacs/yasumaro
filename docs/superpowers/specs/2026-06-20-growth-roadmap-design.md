# Yasumaro 成長ロードマップ設計書

- **日付**: 2026-06-20
- **対象**: Yasumaro Chrome 拡張機能の成長・UX 改善フィードバックを PBI 化
- **アプローチ**: 基盤優先（Foundation First）— v6.1.x で離脱防止基盤を固め、v6.3.x 以降でバイラル・キラー機能を追加

---

## 1. 背景と問題

2026-06-20 に受領したフィードバックでは、Yasumaro の技術的基盤は高く評価されている一方、以下の成長阻害要因が指摘された。

| 軸 | 現状の課題 | 改善の方向性 |
|---|---|---|
| 1. 初期設定 | API キー取得やローカル LLM 構築が必要で、一般ユーザーが離脱 | 対話型ウィザードで必要最小限の設定だけを案内 |
| 2. UX の安心感 | 「勝手に動く」ことへの恐怖、コストの見えない化 | コスト/節約時間の可視化、手動実行の視認性向上 |
| 3. バイラル設計 | ローカル完結ゆえに口コミが外部に漏れにくい | Web Share API 連携、週次レポート生成 |
| 4. ターゲット拡大 | Obsidian ユーザー向けツール感が強い | Markdown コピーで他ノートアプリ対応、ストア掲載情報刷新 |
| 5. キラー機能 | 蓄積データの活用が要約・検索で止まっている | SQLite 内 RAG チャットで「第二の脳」化 |

---

## 2. ゴールと非ゴール

**ゴール**
- 5 軸すべてを独立した PBI（PBI-19 〜 PBI-26）として定義する
- v6.1.x / v6.3.x 以降のリリース計画を明確にする
- 各 PBI の受け入れ基準、依存関係、プライバシー上の考慮を記述する

**非ゴール**
- この設計書ですべての PBI の詳細実装設計を完結させる（各 PBI は個別の計画書を持つ）
- 新しいサーバー/バックエンドの運用（ローカル完結方針を維持）
- 既存の Obsidian 連携機能の削除や弱体化

---

## 3. 制約と前提

| 項目 | 決定 |
|---|---|
| サーバー運用 | 行わない。ローカル完結を維持 |
| お試し AI | 開発者提供プロキシサーバーは採用しない。代わりに設定ウィザードで案内 |
| SNS シェア | Web Share API を使用。拡張機能側からの直接投稿は行わない |
| RAG チャット | ローカル LLM 優先。ユーザー設定のクラウド API も許可（デフォルト OFF） |
| リリース分け | v6.1.x で基盤、v6.3.x 以降でバイラル・キラー機能 |

---

## 4. ロードマップ構成

```
v6.1.x（Foundation）
├── PBI-19: 対話型設定ウィザード
├── PBI-20: 手動実行ボタンの視認性向上
├── PBI-21: Markdown 1クリックコピー（Obsidian 以外対応）
└── PBI-22: Chrome Web Store ランディングページ刷新

v6.3.x 以降（Viral + Killer）
├── PBI-23: コスト・節約時間の可視化

├── PBI-25: 週次インプットレポート
└── PBI-26: RAG チャット（SQLite 内検索＋AI 応答）
```

PBI 番号は既存 PBI（00 〜 18）の次を割り当てる。

## 5. v6.1.x PBI 詳細

### PBI-19: 対話型設定ウィザード

**ユーザーストーリー**
Yasumaro を初めて開いたユーザーとして、自分の用途に合った設定だけをステップで案内してほしい。なぜなら、設定画面の情報量が多すぎて圧倒されるから。

**受け入れ基準**
- 初回ポップアップ表示時にウィザードを表示（既存プライバシー同意フローの直後）
- 3 つのユーザータイプから選択する：
  - 「Obsidian 使い」→ Obsidian 接続設定を案内
  - 「SQLite（Obsidian なし）派」→ AI プロバイダー設定のみ案内
  - 「とりあえず試したい」→ 最小設定で完了
- 各ステップで「スキップ」ボタンを配置する
- ウィザード完了後、ダッシュボードの該当セクションが開いた状態で遷移する
- i18n（ja/en）対応

### PBI-20: 手動実行ボタンの視認性向上

**ユーザーストーリー**
自動記録が不安なユーザーとして、拡張機能アイコンをクリックしたときに「今すぐ要約・保存」ボタンが一目でわかるようにしてほしい。

**受け入れ基準**
- ポップアップのメイン画面で「今すぐ記録」ボタンを視覚的に強調（プライマリーカラー、サイズ拡大）
- コンテキストメニュー（右クリック）から「Yasumaro でこのページを記録」を実行可能にする
- 手動実行時の進捗状態（要約中 / 保存完了）をポップアップ内で表示する
- 完了時のトースト/バッジ通知を強化する

### PBI-21: Markdown 1クリックコピー

**ユーザーストーリー**
Obsidian を使っていないユーザーとして、記録した要約を Notion やメモアプリに簡単に貼り付けたい。

**受け入れ基準**
- ポップアップの記録完了画面に「Markdown をコピー」ボタンを追加する
- ダッシュボードの SQLite History 各行に「コピー」アクションを追加する
- クリップボードにコピーする形式：タイトル / URL / 要約 / タグ / 保存日時を含む Markdown
- コピー成功時にトーストを表示する
- 形式は将来的なカスタマイズを見据え、テンプレート関数として分離する

### PBI-22: Chrome Web Store ランディングページ刷新

**ユーザーストーリー**
非エンジニアの潜在的ユーザーとして、ストアページで Yasumaro の価値が直感的に伝わってほしい。

**受け入れ基準**
- 説明文を「AIがあなた専用のブラウジングデータベースを自動構築」中心に刷新する
- スクリーンショット/GIF を新 UI（和モダンテーマ適用後）で更新する
- 必要に応じてプロモーション動画素材を用意する
- `store-assets/` ディレクトリに整理し、審査提出時の運用を明文化する

## 6. v6.3.x 以降 PBI 詳細

### PBI-23: コスト・節約時間の可視化

**ユーザーストーリー**
ヘビーユーザーとして、Yasumaro が自分にどれだけ価値を生んでいるか（時間節約 / 推定コスト）をダッシュボードで確認したい。

**受け入れ基準**
- ダッシュボードに統計パネルを追加する
- 表示項目：
  - 「今月の推定閲覧時間」
  - 「AI 要約実行回数」
  - 「推定トークン消費量（プロバイダー別集計）」
  - 「手動要約に相当する時間を節約した推定値」
- データソースは SQLite の `browsing_logs` テーブルとする
- 月次リセットではなく、累積＋期間フィルタを提供する

### PBI-24: Web Share API 連携

**ユーザーストーリー**
面白かった記事の要約を、安全に他者と共有したい。

**受け入れ基準**
- ダッシュボード / ポップアップの要約表示に「共有」ボタンを追加する
- ボタン押下で `navigator.share()` を呼び出す
- シェアテキストに `#Yasumaro` ハッシュタグを含める
- デフォルトで「タイトル + URL + 要約の冒頭 + via #Yasumaro」形式とする
- ユーザーが共有する前に、PII が含まれていないかの警告を表示する
- プライバシーモード HIGH の場合はシェア機能を無効化する

### PBI-25: 週次インプットレポート

**ユーザーストーリー**
週末に自分のインプットを振り返りたい。できればそれを画像として SNS にシェアしたい。

**受け入れ基準**
- ダッシュボードに「週次レポート」タブ/セクションを追加する
- 過去 7 日間のデータを集計する：
  - ジャンル円グラフ（タグまたは AI 判定によるカテゴリ分類）
  - 記事数 / 推定閲覧時間
  - AI 生成の「今週の一言サマリー」
- 「画像として保存」ボタンで PNG 出力（Canvas / html-to-image）
- 週次レポートの生成は手動トリガーとする（自動生成は後続検討）

### PBI-26: RAG チャット

**ユーザーストーリー**
溜まった自分の閲覧履歴の中で、過去に読んだ内容について AI に質問したい。

**受け入れ基準**
- ダッシュボードにチャット UI を追加する
- ユーザーの質問を SQLite FTS5 で検索し、関連する閲覧履歴を取得する
- 取得結果＋質問を AI（ユーザー設定のプロバイダー）に送信する
- 回答に参照元 URL を表示する
- 設定で「RAG を有効化」のトグルを設置する（デフォルト OFF）
- ローカル LLM が設定されている場合はローカル優先、なければクラウド API を使用する
- チャット履歴はセッション内（または SQLite 内）に一時保存する

## 7. 依存関係と実装順序

```
v6.1.x 依存関係
├── PBI-22 Store ランディング
│     └─ 独立。UI スクリーンショットは PBI-19〜21 完了後が理想
├── PBI-19 設定ウィザード
│     ├─ 既存のプライバシー同意フロー（privacyConsent.ts）に統合
│     └─ ダッシュボードのパネル開閉ロジックを再利用
├── PBI-20 手動実行ボタン
│     ├─ 既存の handleManualRecord を拡張
│     └─ contextMenus 権限を manifest に追加
└── PBI-21 Markdown コピー
       ├─ 既存の formatEntriesToMarkdown を汎用化
       └─ ポップアップ側にも Markdown 生成ロジックを流用

v6.3.x 依存関係
├── PBI-23 コスト可視化
│     └─ PBI-21 の統計基盤を流用可能
├── PBI-24 Web Share
│     └─ PBI-21 の Markdown テンプレートを流用
├── PBI-25 週次レポート
│     ├─ PBI-23 の集計ロジックを流用
│     └─ html-to-image ライブラリ追加検討
└── PBI-26 RAG チャット
       ├─ SQLite FTS5 全文検索（既存）
       ├─ AI プロバイダー抽象化（既存）
       └─ 新規：コンテキスト構築 + プロンプト設計
```

**実装上の注意**
- PBI-19 は UX 変更が大きいため、v6.1.x の先頭に置く
- PBI-20 は `contextMenus` 権限追加が必要。Chrome Web Store 審査時に正当化が必要
- PBI-26 は技術的リスクが高いため、v6.3.x の先頭に置き、スパイクを含める

## 8. プライバシー・セキュリティ上の考慮

| PBI | リスク | 対策 |
|---|---|---|
| PBI-19 ウィザード | ユーザー選択に応じて必要最小限の権限を取得 | タイプ別に要求権限を明示する |
| PBI-20 コンテキストメニュー | `contextMenus` 権限追加、審査で正当化が必要 | `PERMISSIONS.md` を更新する |
| PBI-21 Markdown コピー | クリップボードに要約が一時的に残る | ローカル処理、外部送信なしを明記する |
| PBI-22 Store ランディング | 説明文と実装の乖離 | PR 時に説明文も更新するフロー化 |
| PBI-23 コスト可視化 | トークン数の推定精度 | 「推定」表記を必須とし、API レスポンスから計算する |
| PBI-24 Web Share | ユーザーが意図せず PII を含む要約を共有 | 共有前確認ダイアログ、HIGH プライバシーモードでは無効化 |
| PBI-25 週次レポート | 画像生成に個人情報が含まれる | ローカル Canvas 処理、SNS 投稿はユーザー操作 |
| PBI-26 RAG チャット | 検索結果を AI に送信する際のデータ漏洩 | デフォルト OFF、ローカル LLM 優先、設定で明示的 ON |

**全 PBI 共通**
- 新機能は既存の PII サニタイズパイプラインを通す
- 外部送信は「ユーザーが選んだ AI プロバイダー」のみとする
- GDPR 準拠のため、新たなデータ保持は最小限にする

## 9. テスト・効果測定方針

### テスト戦略

| PBI | 統合テスト | 単体テスト | E2E / 手動 |
|---|---|---|---|
| PBI-19 | ウィザードのステップ遷移 | タイプ別の設定保存 | 実際の初回起動フロー |
| PBI-20 | handleManualRecord 拡張 | context menu 登録 | 右クリックからの実行 |
| PBI-21 | Markdown 生成関数 | クリップボード書き込み | 各アプリへの貼り付け |
| PBI-22 | — | — | ストアページ更新確認 |
| PBI-23 | 集計クエリ | トークン推定計算 | ダッシュボード表示 |
| PBI-24 | navigator.share 呼び出し | シェアテキスト生成 | 実機での共有シート |
| PBI-25 | レポート生成 | Canvas 画像出力 | 画像保存・シェア |
| PBI-26 | RAG パイプライン | コンテキスト構築 | 実際の質問応答 |

### 効果測定（成功指標）

| 指標 | 測定方法 |
|---|---|
| 初回設定完了率 | ウィザード最後のステップ到達率 |
| 7日間の継続使用率 | インストール後 7 日以内の記録回数 |
| 手動実行回数 | ポップアップ / コンテキストメニュー別 |
| ストア CV 率 | Chrome Web Store の表示→インストール |
| SNS シェア回数 | Web Share API 起動回数（匿名） |
| 週次レポート開封率 | レポートセクションのアクセス数 |
| RAG 利用率 | チャット機能の ON 率・質問数 |

## 10. 未確定・後続検討

- 各 PBI の詳細設計は個別の計画書で実施する
- PBI-26（RAG チャット）は技術的リスクが高いため、正式実装前にスパイクを実施する
- PBI-22（Store ランディング）のプロモーション動画は、PR 動画スキルで別途検討する
- PBI-19 のユーザータイプ分岐は、利用データに基づいて最適化する可能性がある
