# PBI: opfsMigrationV2 除去可否の判断（意思決定PBI）

## 種別

**本PBIは実装PBIではなく「意思決定PBI」である。** ゴールはコード変更ではなく、`opfsMigrationV2.ts` / `opfsMigrationV2Reader.ts` を除去してよいかどうかを判断するための調査と基準策定。

## ユーザーストーリー

開発者として、旧 OPFS DB → 新 DB への一回限り冪等移行コード（`opfsMigrationV2.ts` / `opfsMigrationV2Reader.ts`）をいつ・どんな基準で除去してよいかを判断したい、なぜなら移行完了済みユーザーの割合が不明なまま除去すると、まだ移行していないユーザーのデータ移行が永久に行われなくなるリスクがあるから。

出典: `docs/blog-6_5/offscreen-opfs-sqlite-coexistence-deep-dive.md`「開発者の方向け: すぐに効く改善案」の備考、および [[2026-07-16-06-fix-idb-fallback-subframe7536-migration]] の実装検討中に切り出された論点。
関連ADR: `dev-docs/ADR/2026-06-17-opfs-fts5-coexistence.md`（ADR-014）。

本PBIは元 `pbi/2026-07-16-03-fix-offscreen-opfs-sqlite-coexistence.md`（brainstormingで深掘り後に3+1分割）の一部。関連: [[2026-07-16-06-fix-idb-fallback-subframe7536-migration]]

## なぜ別軸の判断が必要か

[[2026-07-16-06-fix-idb-fallback-subframe7536-migration]]（wa-sqlite依存除去）は技術的な移行作業だが、`opfsMigrationV2` の除去は「どれだけのユーザーが移行済みか」という**利用実態に基づく判断**であり、コードの技術的な移行完了とは独立した意思決定軸である。両者を同一PBIで扱うと、技術移行が終わっても利用実態の確認が終わらない限りマージできない、という不要な結合が生じる。

## 既実装確認（フェーズ0で実施済み）

```bash
grep -n "OPFS_MIGRATION_V2_DONE\|migrateOldOpfsDb\|readOldDbRecords" src/offscreen/opfsWorker.ts
# → L14-15: import migrateOldOpfsDb, readOldDbRecords/deleteOldDbFile
# → L122-158: runMigrationV2() 関数内で二重ガードを実施
```

現状の実装（`src/offscreen/opfsWorker.ts` 122-158行目）:
- `migrationV2AttemptedThisSession`: Worker生存期間中の一度きり実行フラグ（セッション単位）
- `StorageKeys.OPFS_MIGRATION_V2_DONE`: `chrome.storage.local` に保存される永続フラグ
- `chrome.storage.local` が利用できない Worker 環境では、旧OPFSディレクトリの不在をもって「移行不要」とみなすフォールバック戦略も実装済み

この二重ガードの存在自体が「まだ移行未完了のユーザーが存在しうる」という前提でコードが書かれていることを示している。**現時点でこのコードを除去すると、未移行ユーザーのデータが失われる可能性がある。**

## 結論（2026-07-17 意思決定）

**判断: 「計測基盤を先に作る」**

### 根拠

| 調査項目 | 結果 |
|----------|------|
| 経過期間 | 2026-06-17（v5.9.9）から 1 ヶ月。40 リリース経過 |
| 普及率計測手段 | **存在しない**。`OPFS_MIGRATION_V2_DONE` は `chrome.storage.local` に保存されるが、ダッシュボードからも SW からも参照されず、opfsWorker.ts 内で完結している |
| データ損失リスク | 除去すると未移行ユーザーの旧 OPFS DB データが永久に読み取れなくなる。1 ヶ月では長期オフラインユーザーの移行未了リスクを否定できない |

### 採用する計測アプローチ

Yasumaro は外部サーバを持たないプライバシー重視の拡張機能であり、ユーザー横断的な集計テレメトリはアーキテクチャ上存在しない。したがって「計測基盤」とは**ダッシュボード診断パネルでの移行状態可視化**を指す。

具体的には:
1. `opfsWorker.ts` の `runMigrationV2()` に移行試行日時・完了日時を `chrome.storage.local` に記録する処理を追加
2. ダッシュボードの SQLite 診断パネル（または新設の移行状態表示）に以下の情報を表示:
   - `OPFS_MIGRATION_V2_DONE`: 移行完了フラグ（既存）
   - `OPFS_MIGRATION_V2_ATTEMPTED_AT`: 初回試行日時（新設）
   - `OPFS_MIGRATION_V2_COMPLETED_AT`: 完了日時（新設）
   - `OPFS_MIGRATION_V2_RECORD_COUNT`: 移行レコード数（新設）
3. 上記の可視化により、ユーザー自身が移行状態を確認できるようにする

ただし、これでも「全ユーザーの何%が移行済みか」という**集計値は取得できない**。この制約を認識した上で、**時間経過（3〜6 ヶ月）+ ダッシュボード可視化による間接的な状況把握**を以て、最終的な除去判断の材料とする。

### 除去の判断基準（再定義）

以下の両方を満たした時点で除去可能と判断する:
1. **経過期間**: 移行導入から **6 ヶ月以上**（2026-12-17 以降）
2. **可視化**: ダッシュボード診断パネルでの移行状態表示を実装・リリース済み

6 ヶ月という期間の根拠:
- Chrome 拡張の自動更新はブラウザ起動時に実行される。6 ヶ月あれば、通常の利用パターンを持つほぼすべてのユーザーが少なくとも 1 回は拡張機能を起動している
- 長期オフラインユーザーのデータ損失リスクは残るが、6 ヶ月経過していれば許容範囲と判断する
- 集計テレメトリがない以上、時間で判断するのが唯一の現実的な手段

## 実装者向け注記

### 現状コードの確認
（着手前に必ず実行すること）
```bash
grep -n "OPFS_MIGRATION_V2_DONE\|migrationV2AttemptedThisSession" src/offscreen/opfsWorker.ts
grep -rn "OPFS_MIGRATION_V2_DONE" src/utils/storage*.ts src/constants/*.ts 2>/dev/null
```

### 落とし穴
- 「コードがシンプルになるから消したい」という技術的動機だけで除去を判断しないこと。本PBIの目的は利用実態に基づく判断基準を作ることであり、技術的なきれいさは判断材料の一つに過ぎない
- 普及率を計測する手段が存在しない場合、それ自体が「今は除去判断ができない」という結論になりうる。無理に期間だけで判断せず、計測基盤の欠如を明示的に記録すること
