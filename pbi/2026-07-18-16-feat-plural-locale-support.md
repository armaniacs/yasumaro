# PBI: 複数形・数量表現のロケール対応（Chrome i18n plural カテゴリ導入）

## ユーザーストーリー
英語版拡張機能ユーザーとして、件数表示（"1 item" / "2 items" のような複数形）が文法的に正しく表示されてほしい、なぜなら現状のメッセージ定義には複数形カテゴリの区別がなく、機械的な文字列結合になっている可能性があるから

## ビジネス価値
- 英語圏ユーザーへの表示品質向上、プロフェッショナルな印象

## 実装者向け注記（フェーズ0の既実装確認結果）

grep で確認済み:
- `public/_locales/en/messages.json` に `plural` カテゴリ（ICU MessageFormatの `{count, plural, ...}` 相当）は現状使用されていない
- 対象となる件数表示箇所を特定するため、実装前に `grep -rn "件\|items\|records" public/_locales/en/messages.json` で該当メッセージキーを洗い出す必要がある

```bash
# 実装前の必須調査コマンド（対象メッセージキーの特定）
grep -n "count\|items\|records\|件" public/_locales/en/messages.json
grep -rn "getMessage(" src/ --include="*.ts" | grep -i "count\|item\|record"
```

**注意**: Chrome拡張機能の `chrome.i18n` API は ICU MessageFormat の `plural` 構文をネイティブサポートしていない（`messages.json` の `placeholders` 機構のみ）。真の複数形対応には、アプリケーション側で件数に応じた文字列を条件分岐で選択する実装が必要になる可能性が高い。実装着手前に `chrome.i18n.getMessage` の制約を踏まえた設計検討が必要。

## BDD受け入れシナリオ

```gherkin
Scenario: 件数が1件のとき単数形で表示される
  Given 保存済み記事が1件ある状態
  When ダッシュボードに件数を表示する
  Then "1 item" のように単数形で表示される（日本語版は影響なし）

Scenario: 件数が2件以上のとき複数形で表示される
  Given 保存済み記事が5件ある状態
  When ダッシュボードに件数を表示する
  Then "5 items" のように複数形で表示される
```

## 受け入れ基準
- [ ] 件数表示を伴うメッセージキーを全て洗い出し、一覧化する
- [ ] 英語版で単数形/複数形が正しく出し分けられる（`chrome.i18n` の制約上、アプリケーション側の条件分岐実装 or 単数形/複数形2キー管理のいずれかで対応）
- [ ] 日本語版の表示に影響がない（日本語に複数形の区別はないため）

## テスト戦略（t_wadaスタイル）

### E2E（最小限）
- ダッシュボードで件数が1件/複数件の状態を再現し、表示テキストを目視確認

### 統合テスト
- 該当なし

### 単体テスト
- 新設する複数形判定ヘルパー関数（もしくは既存i18nヘルパーの拡張）に対して、件数0/1/2/多数のケースで正しい文字列が返ることを検証

## 実装アプローチ
- **Outside-In**: まず対象メッセージキーの洗い出し（調査タスク）→ 単体テストで複数形判定ロジックをRed→Green→実際のUI箇所への適用

## 見積もり
3pt（対象箇所の洗い出しに時間がかかるため中規模。`chrome.i18n` の制約により素朴な実装では対応できない可能性があり、設計検討を含む）

## 技術的考慮事項
- 依存関係: なし
- テスタビリティ: 複数形判定ロジックは純粋関数として切り出せば単体テスト容易
- 非機能要件: i18n品質

## 落とし穴
- `chrome.i18n.getMessage` はICU pluralをサポートしないため、`messages.json` に `_one` / `_other` のような複数バリアントキーを用意し、呼び出し側で件数に応じてキーを選択する設計になる可能性が高い。実装前にこの設計をチームで合意すること
- 対象箇所が想定より多い場合、1PBIに収まらない可能性がある。洗い出し結果次第でさらに分割を検討する

- [ ] 英語版UIで単数形/複数形の目視確認が完了している
