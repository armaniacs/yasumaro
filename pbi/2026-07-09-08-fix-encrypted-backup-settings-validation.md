# PBI: 暗号化バックアップ復元時の設定値スキーマ検証

## ユーザーストーリー
ユーザーとして、暗号化バックアップをインポートしても改ざんされた設定値（APIキー・接続先URL・PAT）がそのまま復元されないようにしてほしい、なぜなら細工されたバックアップファイルを誤ってインポートした場合でも、AIプロバイダのAPIキーやObsidian接続先、GitHub PATが不正な値に書き換えられる被害を防ぐ必要があるから

## ビジネス価値
- 悪意あるバックアップファイル（他人から渡された・改ざんされた）をインポートした際の資格情報漏洩・不正送信先変更リスクを排除する
- 型検証により壊れたバックアップの復元時にも予測可能なエラーとなり、原因調査が容易になる

## テスト戦略（t_wadaスタイル）

### E2Eテスト
- ダッシュボードから暗号化バックアップをエクスポート→インポートし、非機密設定が復元されAPIキーが変化しないことを確認するシナリオテスト

### 統合テスト
- `importEncryptedBackup()` に対し、正常なpayload / 許可リスト外キーを含むpayload / 型不正な値を含むpayloadを渡し、それぞれの復元結果（どのキーが反映されどのキーがスキップされたか）を検証

### 単体テスト
- 新設する設定値バリデーション関数（例: `validateRestorableSettings()`）の単体テスト
  - 許可リスト内の正常値 → そのまま通過
  - 許可リスト外のキー → 除去される
  - 型不一致（数値期待に文字列など） → 除去またはデフォルトにフォールバック
  - 範囲外の値（負の保持日数など） → 除去またはデフォルトにフォールバック
  - 機密キー（APIキー・PAT等）が payload.settings に含まれていても復元対象にならないこと

## 実装者向け注記

### 現状コードの確認
（着手前に必ず実行すること）
```bash
grep -rn "isBackupPayload\|importEncryptedBackup\|saveSettings(payload" src/
grep -rn "StorageKeys\." src/utils/storage.ts | grep -i "api_key\|pat\|token"
```
`src/utils/storage/types.ts` の `Settings` 型定義と `DEFAULT_SETTINGS` を確認し、機密扱いすべきキー（`*_API_KEY`, `*_TOKEN`, `*_PAT` 系）を洗い出すこと。

### 実装手順
1. `src/utils/storage/types.ts` または新規 `src/dashboard/backupAllowlist.ts` に、バックアップから復元可能な設定キーの許可リストを定義する（例: `RESTORABLE_SETTINGS_KEYS`）
2. 各キーに対応する簡易バリデータ（型ガード + 範囲チェック）を用意する
3. `importEncryptedBackup()` 内で `payload.settings` を許可リスト+バリデータでフィルタリングした `Partial<Settings>` を生成してから `saveSettings()` に渡すよう変更する
4. スキップされたキーをログ（`addLog` 等の既存ロガー）に記録し、戻り値に `skippedKeys` のような情報を含めて呼び出し元（UI）でユーザーに通知できるようにする

### 落とし穴
- `Settings` 型は多数のキーを持つため、許可リストを手動で書くと将来のキー追加時に同期漏れが起きやすい。可能なら「機密キーの除外リスト」方式（デフォルト許可、機密キーのみ拒否）と「許可リスト」方式のどちらが保守しやすいかをチームで検討すること（本PBIでは指摘方針通り許可リスト方式を採用する）
- `saveSettings()` はAPIキーを暗号化する副作用を持つため、機密キーを渡さないようにすることで意図せぬ暗号化処理の実行も避けられる
