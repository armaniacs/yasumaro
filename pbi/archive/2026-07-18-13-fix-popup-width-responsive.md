# PBI: ポップアップ幅の固定値をレスポンシブ対応に変更

## ユーザーストーリー
拡張機能ユーザーとして、様々なディスプレイ・ズーム設定でもポップアップUIが正しく収まってほしい、なぜなら現状は360px固定幅で、環境によってはコンテンツが見切れたり不要な余白が生じるから

## ビジネス価値
- 高DPI環境や画面拡大設定を使うユーザーの表示崩れを防止
- 実装コストが極めて低く即座に対応可能

## 実装者向け注記（フェーズ0の既実装確認結果）

grep・Read で確認済み:
- `entrypoints/popup/styles.css:112` — `body { width: 360px; ... }` で固定幅
- Chrome拡張のpopupは`chrome://extensions`のビューポート制約を受けるため、`max-width: 100vw` を追加しても実害はない

```bash
# 実装前の再確認コマンド
grep -n "width: 360px" entrypoints/popup/styles.css
```

## BDD受け入れシナリオ

```gherkin
Scenario: 通常のディスプレイでポップアップが従来通り表示される
  Given 標準的な解像度・ズーム100%の環境
  When 拡張機能アイコンをクリックしポップアップを開く
  Then ポップアップの幅は360px前後で表示され、レイアウト崩れがない

Scenario: 画面幅が制約される環境でもポップアップがはみ出さない
  Given ブラウザウィンドウの表示可能領域が360pxより狭い環境（極端な高倍率ズーム等）
  When 拡張機能アイコンをクリックしポップアップを開く
  Then ポップアップはビューポート幅を超えず、横スクロールが発生しない
```

## 受け入れ基準
- [ ] `entrypoints/popup/styles.css` の `body` に `min-width: 360px; max-width: 100vw;` を設定（`width: 360px` 固定を置き換え）
- [ ] 既存のポップアップ内レイアウト（タブ、フォーム等）が崩れない

## 落とし穴
- `min-width: 360px` を残さないと、逆に小さすぎる幅で内部レイアウトが崩れる可能性があるため、`max-width: 100vw` の追加のみで `min-width` は必ず残すこと
