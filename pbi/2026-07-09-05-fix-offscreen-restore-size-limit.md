# PBI: SQLite リストア時のバイナリペイロードにサイズ上限を設ける

## ユーザーストーリー
保守開発者として、SQLite リストア（`SQLITE_RESTORE`）で読み込むバイナリペイロードに上限を設けたい。なぜなら現状はサイズ検証がなく、悪意ある巨大ペイロードでストレージクォータを枯渇させる攻撃や、メモリ枯渇を防げないから。

## ビジネス価値
- リストア経路の DoS / リソース枯渇を防ぐ
- 入力検証の欠落というセキュリティ上の弱点を塞ぐ

## 既実装確認（Phase 0）
- `grep -rn "SQLITE_RESTORE" src/offscreen/offscreen.ts` → ハンドラでサイズ検証なしを確認
- **実装状態**: Checking Team Wave 3（2026-07-09）で既に修正済み（100MB 上限）・テスト追加済み（未コミット）。本 PBI は回帰仕様として記録

## BDD受け入れシナリオ

```gherkin
Scenario: 許容サイズ内のペイロードはリストアされる
  Given リストア用バイナリが 50MB である
  When SQLITE_RESTORE メッセージが送信される
  Then リストアが正常に実行される

Scenario: 許容サイズを超えるペイロードは拒否される
  Given リストア用バイナリが 100MB を超えている
  When SQLITE_RESTORE メッセージが送信される
  Then エラーレスポンスが返り、リストアは実行されない
```

## 受け入れ基準
- [ ] `SQLITE_RESTORE` ハンドラで `rawData.length > 100 * 1024 * 1024` の場合はエラー応答を返す
- [ ] 上限内は既存通りリストアされる
- [ ] 既存テスト `offscreen.test.ts` の該当コメント／関連テストが維持される
