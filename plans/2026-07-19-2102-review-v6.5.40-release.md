# Checking Team レビューレポート

## 総合評価: 89/100 (ランク: A)

## 重要指摘事項（優先度順）

### [High] Message type definitions duplicated — `messageTypes.ts` vs `messaging/types.ts`

- 指摘者: **Maintainability Guardian** (High), **System Architect** (Medium), **API & Contract Negotiator** (Medium)
- 場所: `src/background/messageTypes.ts:133-152` および `src/messaging/types.ts:152-167`
- 影響: `ExtensionMessage` (background) と `ServiceWorkerRequest` (messaging) が同一プロトコルを表現しているが、ペイロード構造が異なり独立して進化。`VALID_MESSAGE_TYPES` (18種) と `validTypes` (15種) は既に不一致（Ping / RefreshLocalMarkdownScheduler / ConsentStateChanged / DashboardSqlite が欠落）。修正時に両ファイルの整合を手動で取る必要があり、バグ原因になる。
- 対処: `messageTypes.ts` を唯一の情報源 (SSOT) とする。`messaging/types.ts` からは `import type` で参照し、`ServiceWorkerRequest` の独自定義を削除。`sendServiceWorkerMessage` / `sendFromPopup` の型引数は `ExtensionMessage['type']` で代用可能。

### [High] SessionStore のストレージバックエンド変更（local → session）による互換性リスク

- 指摘者: **Legacy Bridge Architect** (High), **Edge & Mobile Strategist** (High - quota)
- 場所: `src/background/sessionStore.ts:23-26`
- 影響: (1) session ストレージはブラウザ終了時にクリアされ、従来 local ストレージに保持されていた `sw:*` データが消失する。 (2) session ストレージの約1MB制限超過時にメモリフォールバックのみで SW 終了時にデータ消失。(3) `migrateFromLocalStorage()` 成功後は session データ消失時に復元手段がない。(4) SW コンテキストでの `navigator.userAgent` 使用によるモバイル判定失敗。
- 対処: `get()` メソッドに session 消失時に local から再移行する fallback 読み取りパスを追加。`navigator.userAgent` を `chrome.runtime.getPlatformInfo()` に変更。

### [High] Service Worker コンテキストでの `navigator.userAgent` 使用

- 指摘者: **Edge & Mobile Strategist** (High)
- 場所: `src/utils/deviceUtils.ts:9`, `src/background/sqliteClient.ts:80`
- 影響: Service Worker 実行時は `navigator` が存在しないか `userAgent` が空文字になるケースがあり、モバイル判定がサイレントに失敗する。
- 対処: `chrome.runtime.getPlatformInfo()` など確実なAPIで判定する。

### [High] chrome.storage.session の 1MB クォータ制約がモバイルで顕在化しうる

- 指摘者: **Edge & Mobile Strategist** (High)
- 場所: `src/background/sessionStore.ts:127-138`
- 影響: SessionStore は session storage の約1MB制限に依存。`sw:tabCache` に多数のタブ情報が蓄積すると超過し、クォータ超過時のフォールバックはメモリ内保持のみで SW 終了時にデータ消失。
- 対処: クォータ超過時に一部キャッシュのみ session に保存する段階的戦略、または `session.set` 前にサイズ推定を行う。

### [Medium] Protocol version mismatch does not reject messages

- 指摘者: **Red Team Leader**
- 場所: `src/background/service-worker.ts:444-451`
- 影響: Service Worker 更新後も古い Content Script が残存するタブでメッセージが処理され続ける。
- 対処: バージョン不一致時にメッセージを拒否する。

### [Medium] 大部分のメッセージハンドラで sender.id 検証が欠如

- 指摘者: **Blue Team Leader**
- 場所: `src/background/handlers/messageHandlers.ts` 全般
- 影響: content script が不正ページに乗っ取られた場合などに、記録処理ハンドラが悪用される可能性。
- 対処: 全ハンドラのエントリポイントで `if (sender.id !== chrome.runtime.id)` の防御的チェックを追加。

### [Medium] `logCritical` が生の details を PII サニタイズなしで console.error に出力

- 指摘者: **Blue Team Leader**
- 場所: `src/utils/logger.ts:701`
- 影響: 暗号化失敗やデータ損失などの重大エラー時、APIキー・URL・個人データがブラウザコンソールに露出する。
- 対処: console.error に渡す前に `sanitizeLogDetails(details)` を通す。

### [Medium] i18n.ts モジュールレベル副作用による非UIコンテキストでのインポート障害リスク

- 指摘者: **System Architect** (Medium), **Refactoring Evangelist** (Low), **Legacy Bridge** (Medium)
- 場所: `src/utils/i18n.ts:185-197`
- 影響: Service Worker や Offscreen Document など `document` が存在しないコンテキストで import すると ReferenceError。
- 対処: DOM依存関数を `i18n-dom.ts` に分離し、モジュールスコープの即時実行を削除する。

### [Medium] SessionStore.waitForFlush() のポーリングによる同期処理が非決定的

- 指摘者: **System Architect** (Medium), **Tuning Expert** (Medium)
- 場所: `src/background/sessionStore.ts:159-176`
- 影響: 10ms 固定ポーリングでフラッシュ完了を検知するが、条件式が不正確でタイミングウィンドウが存在。
- 対処: `flushPromise` をキュー管理し、`Promise.all(this.flushQueue)` を返す設計に変更。

### [Medium] Content Script extractor のグローバル可変状態と設定読み込みの反復

- 指摘者: **Maintainability Guardian**
- 場所: `src/content/extractor.ts:46-103`
- 影響: AI要約クレンジングオプションが増えるたびに3箇所の変更が必要。設定値が25個に達し、拡張性とテスタビリティを損なう。
- 対処: `CleansingConfig` インターフェースに集約し、ループ処理で一度に読み込む。

### [Medium] メッセージハンドラ内での RecordingPipeline 構築コードの重複

- 指摘者: **Maintainability Guardian**
- 場所: `src/background/handlers/messageHandlers.ts:300-306, 353-359`
- 影響: 2つのハンドラが `createRecordingPipeline` ファクトリを使わず同一コードを重複。`as any` キャストも使用。
- 対処: `createRecordingPipeline(deps)` ファクトリをハンドラ側で使用し、`as any` キャストを排除する。

### [Medium] Logger 保持期間の短縮（7日→3日、1000件→500件）

### [Medium] `entrypoints/options/i18n.ts` 削除と統合による型の非互換

### [Medium] optimisticLock.ts: CAS が真のアトミック性を持たない（TOCTOU）

- 指摘者: **Data Integrity Expert**
- 場所: `optimisticLock.ts:134-183`
- 影響: write と re-verify の間に他プロセスが割り込むと両方のライターが成功と判定されるレース。
- 対処: 現実的なリスクは低い。将来 `chrome.storage.session` の single-writer セマンティクスを活用するか、専用 Mutex を導入。

### [Medium] savedUrlStore: デュアルライトの非トランザクション性

- 指摘者: **Data Integrity Expert**
- 場所: `savedUrlStore.ts:109-146`
- 影響: 互換性用キーの書き込み失敗時に両キーの不整合が発生。コード内で自己言及済み。
- 対処: SQLite が信頼できるストレージになった現在、互換性キーの書き込みを削除するタイミングを検討。

### [Medium] settingsStore.ts: getSettings に重複した移行/復号ロジック

- 場所: `settingsStore.ts:234-253, 314-347`

- 対処: `_applyMigrationsAndDecrypt()` ヘルパーに抽出。
