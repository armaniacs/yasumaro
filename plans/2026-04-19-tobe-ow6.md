## 3. Svelte + Tailwind導入 ⏸ 見送り（popup復元）

- **目標:** UI保守性の向上とCSS干渉問題の解決
- **結果:** popupでの適用は失敗し、元のvanilla UIに復元

- **経緯:**
  1. **2026-04-18〜21:** `entrypoints/popup/` にSvelte 5 + Tailwindで新規popup UIを構築
     - App.svelte（4タブ: General/Domain/Prompt/Privacy）
     - Svelte 5 `$state()` リアクティブ変数
     - Build成功、chrome.storage連携も動作
  2. **2026-04-22 動作確認:** 実際のChrome拡張機能でpopupを開くと、元の入り組んだUI（ステータスパネル、現在ページ情報、バナー、詳細設定等）が完全に消失
     - 原因: 元のpopupは814行の複雑なHTML。Svelte化では単純な設定フォームのみ再現し、既存機能をカバーできていなかった
  3. **復元決定:** 元のpopup UI（`src/popup/popup.html` + `popup.ts`）を `entrypoints/popup/` に復元。Svelte関連ファイルを削除

- **削除したファイル:**
  - `src/popup/App.svelte`
  - `src/popup/components/Common/Button.svelte`
  - `src/popup/components/Common/Input.svelte`
  - `src/popup/components/Settings/ProviderSelect.svelte`
  - `src/popup/components/DomainFilter/FilterMode.svelte`
  - `src/popup/components/DomainFilter/DomainList.svelte`
  - `src/popup/components/Navigation/TabList.svelte`
  - `src/popup/styles/app.css`

- **教訓:**
  - 既存の複雑なUIをフレームワークに移行する際は、機能同等性を先に検証すべき
  - popupのような小さな画面でも、814行のHTMLが持つ情報量を再現するには相当の工数が必要
  - 「Buildが通る」「設定が保存できる」は必要条件だが、十分条件ではない

- **今後の展望:**
  - dashboard（options）やoffscreen等、新規画面・小規模UIでのSvelte+Tailwind適用を再検討
  - popupについては、現行のvanilla UI + CSS Variables方式を維持

## 備考 / Notes

- **Safari対応について**
  - Apple Developer Program（年間99ドル）加入が必要のため、今回は見送り
  - 将来的に余裕があれば再検討

- **パフォーマンステストについて**
  - ベンチマークやプロバイダー別SLA監視は難易度が高いため見送り
  - 実需発生時に再検討

- **キーボード操作対応について**
  - 本拡張は「じっくり読んだWebページを記録」が本質
  - ユーザーがスクロールして読んだページを保存するため、キーボード操作は必須でない
  - ADRとして記録：「キーボード操作対応は実施しない」

- **スクリーンリーダー対応について**
  - 実装難易度とユーザー層を考慮し、今回は見送り

- **popup UI復元に関する補足 (2026-04-22)**
  - 当初「Svelte + Tailwindでpopup UIを刷新する」計画だったが、実際のChrome拡張機能で動作確認したところ、元のv5.1.4の入り組んだUIが完全に消失していた
  - 元のpopup UIは814行のHTMLで、ステータスパネル（現在ページのドメイン/プライバシー/キャッシュ/最終保存/クレンジング/Trustレベル）、バナー、詳細設定などを含む
  - Svelteで再現するには数週間の工数が必要と判断し、既存ユーザーへの影響を避けるため元のUIを復元
  - 復元後、`npm run validate` 2267 tests passed、Buildも正常に完了
  - 技術的負債として「popup UIのCSSが2,377行（styles.css）」という課題は残存するが、機能優先で保留
