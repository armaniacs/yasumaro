# PBI: CHANGELOG リリースノートガイドラインの改善

元指摘: Checking Team (Medium: Documentation Architect)

## ユーザーストーリー
開発チームとして、CHANGELOG.md の連日リリースが続く場合に、各リリースの意図（hotfix / 即時反映 / 新機能）が読み手に伝わるように改善したい、なぜなら v6.5.39（7/18）→ v6.5.40（7/19）のように連日リリースが発生した場合、バグフィックス版と新機能版の区別が読み手に伝わらず、不安感を与える可能性があるから

## ビジネス価値
- リリースノートの透明性向上
- ユーザー・開発者の混乱防止
- リリースポリシーの明確化

## BDD受け入れシナリオ

```gherkin
Feature: リリースノートガイドライン

  Scenario: 連日リリースの場合に意図が明記される
    Given 2日連続でリリースが発生する
    When 2つ目のリリースノートを生成する
    Then リリースノートの冒頭に「このリリースは hotfix です」「このリリースは前日のレビュー指摘修正です」等のコンテキストが含まれる

  Scenario: 通常リリースの場合は追加文言がない
    Given 間隔が空いたリリース
    When リリースノートを生成する
    Then 通常のフォーマットで出力される
```

## 受け入れ基準
- [ ] CHANGELOG.md 冒頭のバージョニングポリシーセクションに「リリース頻度・連日リリース時のガイドライン」を追記
- [ ] 連日リリース時のテンプレート文言を作成
- [ ] コミット済み

## テスト戦略
- 該当なし（ドキュメントのみの変更）

## 実装アプローチ
- CHANGELOG.md のバージョニングポリシーセクションに以下の内容を追記:
  - 連日リリースが発生するケース（hotfix、レビュー指摘の即時反映、CI/pipeline修正）
  - 各リリースエントリの先頭にリリース種別を明記するスタイル

## 見積もり
1pt（ドキュメント追記のみ）

## 技術的考慮事項
- この変更は開発プロセスのガイドラインであり、コードへの影響はない
- `generate-release-notes.js` は CHANGELOG.md のエントリをそのまま抽出するため、リリースノート生成に影響しない

## 落とし穴
- 特にありません
