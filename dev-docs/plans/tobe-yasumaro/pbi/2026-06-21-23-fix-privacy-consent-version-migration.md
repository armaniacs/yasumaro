# PBI-23: プライバシー同意バージョン移行

## ユーザーストーリー

GDPR 準拠の拡張機能ユーザーとして、プライバシーポリシーが更新された際に再同意を求められたい。なぜなら、旧形式（boolean）の同意データではバージョン管理がされておらず、ポリシー変更後も古い同意が有効なままになるから。

## ビジネス価値

- GDPR 第 7 条（同意の撤回）への準拠を維持
- プライバシーポリシー更新時の再同意フローを信頼性あるものにする
- 既存ユーザーの同意データを失わずに新形式へ移行

## BDD 受け入れシナリオ

```gherkin
Scenario: 旧形式（boolean）の同意データを新形式に移行する
  Given ユーザーが v5.x でプライバシー同意を「true」（boolean）として保存している
  When v6.3.x にアップグレードして拡張機能が起動する
  Then 同意データが `{ version: 1, consented: true, timestamp: ... }` のオブジェクト形式に変換される
  And 再同意モーダルは表示されない

Scenario: プライバシーポリシーが更新されて再同意が必要な場合
  Given ユーザーが同意バージョン 1 で同意済みである
  When プライバシーポリシーがバージョン 2 に更新される
  And ユーザーが拡張機能を開く
  Then プライバシー同意モーダルが再表示される
  And ユーザーが同意すると同意バージョンが 2 に更新される

Scenario: 同意を拒否した場合の動作
  Given ユーザーが再同意モーダルで「拒否」を選択した
  When 3 回拒否する
  Then 制限モードで起動する（AI サマリー・Obsidian 保存・SQLite 保存が無効化）
  And 30 日後に再表示される
```

## 受け入れ基準

- [ ] `StorageKeys.PRIVACY_CONSENT` のデータ形式を `boolean` → `{ version: number, consented: boolean, timestamp: number }` に拡張
- [ ] 起動時に旧形式（boolean）を検出し、新形式へ自動変換するマイグレーションロジックを実装
- [ ] `PRIVACY_CONSENT_VERSION` 定数を追加し、ポリシー更新時にバージョン番号をインクリメント
- [ ] 同意バージョンが現在のバージョンより古い場合、再同意モーダルを表示
- [ ] 既存の 3 回拒否→制限モードの動作を維持
- [ ] テストで旧形式→新形式の移行を保証

## テスト戦略（t_wada スタイル）

### 単体テスト
- `boolean` → オブジェクト形式への変換が正しく行われること
- 既にオブジェクト形式の場合は変換がスキップされること
- 同意バージョンの比較ロジック
- 3 回拒否カウンターの動作

### 統合テスト
- 起動時のマイグレーションフロー
- 再同意モーダルの表示条件

## 実装アプローチ

- `src/popup/privacyConsent.ts` の `initializePrivacyConsent()` にマイグレーションロジックを追加
- `StorageKeys.PRIVACY_CONSENT_VERSION` を `src/utils/storage/types.ts` に追加
- 既存の `PRIVACY_CONSENT_LAST_DENIAL_TIME` と組み合わせて使用
