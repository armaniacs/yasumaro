ウェブページをAIに要約させるとき、こんなことを考えたことはないでしょうか。

「このページのコンテンツ、半分以上は本文じゃないんじゃないか」

実際、モダンなウェブページには本文以外の要素が大量に含まれています。ネットバンキングやSaaSのページなら残高表示やログインフォーム。ニュースサイトならクッキー同意バナー、関連記事、ニュースレター購読フォーム、SNSシェアボタン、著者プロフィール——これらすべてがAIへのプロンプトに混入していくわけです。

この問題には2つの側面があります。

ひとつは**プライバシー**。口座番号や残高のようなデータがそのままAIプロバイダのAPIに飛んでいくのは避けたいです。

もうひとつは**精度**。ノイズが多いと要約が散漫になります。「このページは何のページですか」と聞いたとき、AIがナビゲーションメニューの文字列をもとに回答するようでは困ります。

Obsidian Weave には、この問題に対処するための2つのクレンジング機能があります。

**スクリーンショット説明**: ポップアップメニューの「詳細を表示」パネル。クレンジング対象件数（フォーム入力等 14件削除、キーワード 6件削除、合計 20件削除）が表示されている。

## 2種類のクレンジング

| 機能 | 目的 | タイミング |
|------|------|------------|
| **Content Cleansing** | Obsidian保存前に不要な情報を削除 | コンテンツ抽出後、Obsidian保存前 |
| **AI Summary Cleansing** | AI要約前に不要な情報を削除 | AI要約直前 |

2つは独立してオン/オフを切り替えられます。ただし、データとしては直列パイプラインです。Content Cleansingで削除された要素は、その後のAI Summary Cleansingには存在しません。元のウェブページへの影響はありません。

## Content Cleansing — 保存前の掃除

Content Cleansingは、Obsidianに保存するコンテンツからノイズを取り除きます。削除方法は2種類あります。

**Hard Strip**は、タグ名や属性で機械的に除外します。対象は固定です。

| 除外対象 | 理由 |
|---------|------|
| `<input>`, `<textarea>`, `<select>` | フォーム入力値 |
| `<button>`, `<form>` | UI操作要素 |
| `<script>`, `<style>`, `<iframe>` | 非コンテンツ要素 |
| `<canvas>`, `<embed>` | メディア要素 |
| `type="password"`, `type="hidden"` | 機密・非表示フィールド |
| `autocomplete` 属性を持つ要素 | 入力補完フィールド |

**Keyword Strip**は、要素のIDやクラス名にキーワードが含まれていたら削除します。デフォルトのキーワードは以下の通りです。ユーザーが追加・変更できます。

| カテゴリ | キーワード |
|---------|----------|
| 残高・資産 | `balance`、`zandaka`（残高） |
| アカウント | `account`、`login`、`password` |
| 明細・取引 | `meisai`（明細）、`rireki`（履歴）、`torihiki`（取引）、`transaction` |
| カード・番号 | `card-number` |
| 支払い・請求 | `payment`、`billing`、`invoice`、`receipt` |
| 契約・保険 | `keiyaku`（契約）、`hoken`（保険） |
| 個人情報 | `address` |

### 記録前に確認できる

ポップアップを開いて「詳細を表示」を押すと、クレンジング対象の件数を記録前に確認できます。

```text
クレンジング
  フォーム入力等: 14件削除
  キーワード: 6件削除
  合計: 20件削除
```

記録ボタンを押す前の段階でページ上の対象要素数が分かるので、「このページは本当に記録していいか」を判断する材料になります。クレンジングが無効でも件数は表示されます。「こんなに削除対象があるのか」と気づいたタイミングでCleansingを有効にすることもできます。

### 記録履歴のフィルタ

記録履歴の画面に「🧹 クレンジング」フィルタが追加されました。クレンジングが実行されたページだけを表示できます。

- `🧹 Hard` — Hard Stripのみが実行された
- `🧹 Keyword` — Keyword Stripのみが実行された
- `🧹 Both` — 両方が実行された

### マスクとの違い

混同しやすいので整理します。

| | マスク（PII Masking） | クレンジング（Content Cleansing） |
|---|---|---|
| 目的 | 個人情報を伏字にする | ノイズ要素を削除する |
| 対象 | メールアドレス・電話番号・クレジットカード番号など | フォーム・スクリプト・特定クラスの要素 |
| 処理 | `****` に置換（元テキストは保持） | DOMから物理削除 |
| タイミング | PrivacyPipeline内（AI送信直前） | コンテンツ取得時（extractor.ts） |
| 記録 | `maskedCount`（件数） | `cleansedReason`（方式） |

両方が同時に動くこともあります。Cleansingでフォーム要素を削除したあと、PIIマスクが残ったテキストをさらに処理します。

## AI Summary Cleansing — AI要約前の掃除

AI Summary Cleansingは、AIへ送るコンテンツからノイズを取り除き、要約精度とトークン効率を上げます。

通常の設定では、広告・ナビゲーション・ソーシャルウィジェットといった「明らかに不要なもの」を取り除きます。さらに一歩踏み込みたい場合は**ディープクレンジング**が使えます。

### ディープクレンジングとは

ディープクレンジングはデフォルト無効の実験的機能です。通常のAI Summary Cleansingが「広告・ナビゲーション・ソーシャルウィジェット」といった明らかなノイズを取り除くのに対し、ディープクレンジングはそこからさらに踏み込みます。

追加される削除対象は大きく3グループです。

**UIコンポーネント**: `button`, `input`, `select`, `details` タグ。フォーム要素やアコーディオンUIは本文テキストに寄与しません。

**見えていない要素**: `[hidden]`, `[aria-hidden="true"]`, `[style*="display:none"]` で非表示になっている要素。ユーザーには見えていないものをAIへのプロンプトに含める理由はありません。

**空の要素**: テキストコンテンツを持たない `div`, `span`, `p`。JavaScriptで動的に生成されたが中身のないコンテナや、スタイル用のシム要素がここに該当します。

さらに、リンク密度フィルタがあります。`ul/ol` リストのなかで70%以上がリンクで構成されているものはナビゲーション的なリストと判定して削除します。本文中の箇条書きを誤って削除しないよう、「テキストの大部分がリンクか」という基準で判定しています。

### 日本語サイト固有のパターン

日本語サイトでよく見られるBEMスタイルのクラス名にも対応しています。

```text
l-footer, l-header, l-sidebar
p-entry__footer, p-entry__header
c-button, c-label
common-footer, sub-column
```

`l-` がレイアウト、`p-` がページコンポーネント、`c-` が共通コンポーネントを意味するBEMバリアントです。これらのクラスは本文ではなく構造的な要素であることが多いため、削除対象にしています。

### なぜデフォルト無効なのか

誤検知のリスクがあるからです。著者情報（`author`, `byline`）の削除は、学術論文や技術ブログでは「誰が書いたか」が文脈として重要なことがあります。空要素の削除も、SSRの段階では空だが後からJavaScriptで内容が挿入される要素が引っかかる可能性があります。

ニュースサイトや商業サイトを多く閲覧する場合は、有効にすることで大幅なノイズ削減が期待できます。使用するサイトの傾向に合わせて判断してください。

## クレンジングを「数値で見る」

処理前後のバイト数を追跡する機能を追加しました。ダッシュボードの記録履歴で確認できます。

```text
コンテンツクレンジング
  元のバイト数: 45,230 bytes
  クレンジング後: 38,410 bytes  (-6,820 bytes, -15.1%)

AI Summary Cleansing
  元のバイト数: 38,410 bytes
  クレンジング後: 12,350 bytes  (-26,060 bytes, -67.9%)
```

バイト数が変化しなかった場合は表示されません。削減がゼロなら見せる意味がないからです。

Q. 2段階でバイト数が表示されているけど、なぜ単位が違うの？

A. 大丈夫です。順番に整理します。Content Cleansingの `originalBytes`/`cleansedBytes` は**テキストベース**で計測しています。Obsidianに保存されるのはテキストなので、実際に保存される量で測るのが自然です。一方、AI Summary Cleansingの `aiSummaryOriginalBytes`/`aiSummaryCleansedBytes` は**outerHTMLベース**です。HTML構造全体を処理するため、outerHTMLで計測するほうが削減効果を正確に反映します。2つの数値を直接比較することはできません。それぞれ別の「尺度」で測っています。

## AIトークン追跡

AIプロバイダーとの通信で使用したトークン数も記録するようにしました。ダッシュボードの履歴エントリを展開すると以下のような形式で表示されます：

```text
トークン数: 送信: 7,398, 受信: 606, 処理時間 3.1秒 (AI: openai / gpt-4o-mini)
```

送信トークン数・受信トークン数のほか、AI処理にかかった時間と、使用したプロバイダー・モデル名も記録されます。AI Summary Cleansingを使った場合は削減率も確認できます。

「ディープクレンジングを有効にすることでトークンを何%削減できるか」を実際のデータで確認できます。クレンジング設定のチューニングに役立ちます。

### AI Summary Cleansingの「理由」詳細

複数種類の要素が削除された場合、記録履歴の「AI要約クレンジング」欄に上位3件の理由が表示されます：

```text
AI要約クレンジング — 1,585要素削除, 理由: ナビゲーション, 広告, ソーシャル
```

理由が1種類のみであれば「ナビゲーション」「広告」などのラベルが単独で表示されます。

## 処理パイプラインの全体像

```text
0. ページ全体のバイト数を記録（document.body.outerHTML）
   ↓
1. コンテンツ抽出（article/main タグから候補要素を選択）
   ↓
2. クローン作成（どちらかのクレンジングが有効な場合のみ）
   ↓
3. Content Cleansing → originalBytes / cleansedBytes を記録
   ↓
4. AI Summary Cleansing → aiSummaryOriginalBytes / aiSummaryCleansedBytes を記録
   ↓
5. AI要約
   ↓
6. Obsidian保存
```

ステップ3と4は**同じクローン**に対して順次実行されます。「設定は独立して有効/無効を切り替えられる、データは直列パイプラインで処理される」という関係です。

## 設定場所

| 機能 | 設定場所 |
|------|---------|
| Content Cleansing（Hard Strip / Keyword Strip） | Dashboard → Content Cleansing タブ |
| AI Summary Cleansing / ディープクレンジング | Dashboard → AI Summary Cleansing タブ |

どちらもデフォルトはオフです。プライバシーに敏感なサイトを多く記録するなら Hard Strip から、AI要約のノイズを減らしたいなら AI Summary Cleansing から試してみてください。ディープクレンジングはその名の通り攻めた設定なので、有効にする場合は履歴パネルで削減量を確認しながら調整するのがよいです。
