# Obsidian Weave v5.0 リリース — コンプライアンス・プライバシー観点からの変更まとめ

> 対象読者: プライバシー管理者、セキュリティ担当者、コンプライアンス審査者、DPO

---

## 概要

Obsidian Weave は Chrome 拡張機能として動作する閲覧履歴記録ツールです。
v4.2.0 → v5.0（内部バージョン v4.10.x）の期間に、**GDPR/CCPA 対応・権限モデルの刷新・セキュリティ修正**が複数実施されました。本稿ではコンプライアンスおよびプライバシー観点で重要な変更を整理します。

---

## 1. GDPR / CCPA 対応の追加（v4.10.5）

### 同意撤回（GDPR Article 7）

`withdrawPrivacyConsent()` 関数を新設し、ユーザーがいつでもデータ収集への同意を撤回できる仕組みを実装しました。撤回操作はダッシュボードの「データ管理」セクションから実行可能で、撤回日時は内部ログに記録されます。

### データ削除権（GDPR Article 17 / CCPA）

ダッシュボードに「全データ削除」ボタンを追加。`chrome.storage.local.clear()` によりローカル保存された全設定・履歴データを削除します。

:::message
**確認ポイント**: 削除は Chrome ローカルストレージに限定されます。外部サーバーへのデータ送信は行っていないため、削除は単一デバイス上の操作で完結します。
:::

---

## 2. 権限モデルの刷新（v4.9.0）

### `<all_urls>` 権限の削除

従来、すべての URL に対するホスト権限（`<all_urls>`）を静的に要求していました。これは Chrome Web Store の審査基準を満たさない可能性があるため、**オプション権限（`optional_host_permissions`）へ移行**しました。

| 変更前 | 変更後 |
|--------|--------|
| インストール時に全 URL へのアクセスを一括付与 | サイトごとにユーザーが明示的に許可 |
| `host_permissions: ["<all_urls>"]` | `optional_host_permissions` へ移動 |

### 影響範囲

- 未許可のドメインでは記録が行われなくなり、`LOCKED` ステータスが表示されます
- 拒否されたドメインの訪問記録は 90 日後に自動削除されます（プライバシー配慮）
- ダッシュボードで頻繁に訪問しているが未許可のドメインを一括確認・許可できます

## 3. セキュリティ修正（v4.10.4 / v4.10.7）

### XSS 脆弱性の修正（v4.10.4）

`trustSettings.ts` における `innerHTML` の使用を `createElement + textContent` に置換。ユーザーがドメイン・TLD を入力する際に悪意のあるスクリプトが注入されるリスクを排除しました。

### プロンプトインジェクション対策の強化（v4.10.7）

AI へ送信するコンテンツに対するサニタイザー（`promptSanitizer.ts`）を精緻化：

- 誤検知率: **80% → 0%** に低減
- 悪意あるプロンプト命令の検出率: **100%** (4/4)
- `SAFE_CONTEXT_PATTERNS` による文脈判定で正当なコンテンツを誤ブロックしない設計

### マスターパスワード無効化時のデータクリーンアップ（v4.10.7）

マスターパスワードを無効化した際、暗号化された API キーを空文字列で上書き。データ残存によるセキュリティリスクを排除します。

---

## 4. Content Cleansing — AI送信前のデータ最小化（v5.0）

### 概要

**Content Cleansing** は、ページコンテンツを AI プロバイダーへ送信する前に、機密性の高い要素を自動削除する機能です。GDPR データ最小化原則（Article 5(1)(c)）への対応として有効な手段となります。

### Hard Strip

タグ名・属性で機械的に除外します:

| 除外対象 | 理由 |
|---------|------|
| `<input>`, `<textarea>`, `<select>` | フォーム入力値 |
| `type="password"`, `type="hidden"` | 機密・非表示フィールド |
| `<script>`, `<style>`, `<iframe>` | 非コンテンツ要素 |
| `autocomplete` 属性を持つ要素 | 入力補完フィールド |

### Keyword Strip

ID/クラス属性に `balance`、`account`、`meisai`（明細）、`card-number`、`payment` 等のキーワードが含まれる要素を削除します。キーワードリストはユーザーが追加・変更可能です。

:::message
**確認ポイント**: Content Cleansing はデフォルト無効です。ネットバンキング・SaaS など機密情報を含むサイトを記録する場合は、ダッシュボードの「Content Cleansing」パネルで有効化することを推奨します。
:::

### AI Summary Cleansing / Deep Cleansing

さらに、AI へ送るコンテンツからナビゲーション・広告要素を除去する **AI Summary Cleansing** と、非表示要素・空要素まで削除する実験的な **Deep Cleansing** も提供しています。ダッシュボード履歴パネルでクレンジング前後のバイト数削減率を確認できます。

## 5. CSP（Content Security Policy）の刷新（v4.10.0 / v4.10.1）

### 接続先ドメインの削減

AI プロバイダーとの接続先を **29 ドメイン → 10 ドメイン（デフォルト）** に削減。残りの 28 プロバイダーはユーザーが明示的に有効化しない限り接続されません。

### 二層 CSP モデル

1. **第一層**: `manifest.json` の `connect-src`（ブラウザレベルで強制）
2. **第二層**: `CSPValidator` クラスによる実行時 URL 検証

この設計により、設定変更があっても CSP が二重にチェックされます（ADR: `0002-csp-layered-security.md` 参照）。

---

## 6. データ整合性の強化（v4.10.4 / v4.10.5）

- **楽観的ロック**: Trust Database への同時書き込みを `withOptimisticLock()` で保護
- **マイグレーションロールバック**: チェックサム検証を追加し、ロールバック失敗時は例外を送出
- **Bloom Filter 整合性チェック**: ハッシュ検証によるデータ改ざん検出

## 注意事項

- 本ツールはすべてのデータを **ローカルの Chrome ストレージ** に保存します。外部サーバーへの自動送信は行いません
- AI 要約機能を利用する場合、ページコンテンツが設定済みの AI プロバイダーへ送信されます
- API キーは PBKDF2 + AES-GCM で暗号化されてローカルに保存されます
