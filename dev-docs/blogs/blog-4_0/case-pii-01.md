# 「🔒マスクあり」と「プライベートページ検出」は別物 — 混同しやすいプライバシー保護の2つの仕組み

ダッシュボードの記録履歴を見ていたら、こんな表示がありました。

```
自動  🔒 5件マスク
https://www.keishicho.metro.tokyo.lg.jp/...
2026/2/23 10:55:44
```

設定は「自動保存時にプライバシー懸念がある場合は保存するかどうか確認する」にしてあったのに、確認ダイアログも通知も出なかった。なぜ保存されてしまったのか？

結論から言うと、**これは正常な動作**でした。2つの異なるプライバシー保護の仕組みを混同していたんです。

---

## Obsidian Smart History の2層プライバシー保護

Obsidian Smart History には、独立した2つのプライバシー保護機能があります。

| 機能 | 何を見るか | いつ動くか | 設定場所 |
|------|-----------|-----------|---------|
| **PIIマスク（🔒）** | ページのテキスト内容（電話番号・メールアドレスなど） | AI送信の直前 | プライバシーモード（Mode A〜D） |
| **プライベートページ検出** | HTTPレスポンスヘッダー | ページ読み込み時 | 自動保存時の確認設定 |

まったく別のレイヤーで動いています。どちらか一方だけが発動することもあれば、両方同時に発動することもあります。

---

## どの保護が発動するかを決めるフロー

```
ページにアクセス
    │
    ▼
[プライベートページ検出] ← HTTPヘッダーを確認
    │
    ├─ Cache-Control: private / Set-Cookie / Authorization あり？
    │       ↓ YES
    │   → 設定に応じて通知・スキップ・そのまま保存
    │
    └─ ヘッダーなし → 通常通り記録処理へ
            │
            ▼
        [PIIマスク] ← ページのテキスト内容を確認
            │
            ├─ 電話番号・メールアドレスなど検出？
            │       ↓ YES
            │   → [MASKED:phoneJp] などに置換してからAIへ送信
            │   → 🔒バッジを記録に残す
            │
            └─ 検出なし → そのままAIへ送信
```

2つの検査はそれぞれ独立したトリガーで動きます。プライベートページ検出はページ読み込み時にHTTPヘッダーを見ます。PIIマスクはその後の記録処理フロー（PrivacyPipeline）の中でAI送信直前に実行されます。「プライベートページかどうか」に関係なく、PIIマスクは必ず走ります。

---

## PIIマスク — テキストの中身を守る

「🔒 5件マスク」バッジが示すのは、ページのテキスト内容にPII（個人を特定できる情報）が含まれていたため、AIに送る前にマスクした、という記録です。

### 検出対象

| 種別 | 検出例 | 置換後 |
|------|--------|--------|
| 電話番号（日本形式） | `03-1234-5678` | `[MASKED:phoneJp]` |
| メールアドレス | `info@example.com` | `[MASKED:email]` |
| クレジットカード番号 | `4111-1111-1111-1111` | `[MASKED:creditCard]` |
| マイナンバー（12桁） | `1234 5678 9012` | `[MASKED:myNumber]` |
| 銀行口座番号 | `1234567` | `[MASKED:bankAccount]` |

警視庁の行政通知ページには、担当部署の電話番号や担当者情報が記載されています。それらが `[MASKED:phoneJp]` のように置き換えられてからAIに渡されたということです。**ページの保存可否とは無関係に、AI送信前に自動適用されます。**

処理には上限があります。ページのテキストが **64KB（約6万5千文字）を超える場合**、PIIマスクはスキップされ生データがAIに渡されます。また、PII検出件数が1000件を超えるか、処理が5秒以上かかる場合も同様です。非常に長いページでは 🔒バッジが出ないケースがあることを覚えておいてください。

### 4つのプライバシーモード

PIIマスクの挙動は、ダッシュボード → Privacy タブのプライバシーモード設定で変わります。

| モード | 動作 | ステータス |
|--------|------|-----------|
| **A: Local Only** | 完全ローカル処理（ブラウザのAIを使用） | 🚧 開発中 |
| **B: Full Pipeline** | ローカル要約 + クラウド仕上げ | 🚧 開発中 |
| **C: Masked Cloud** | PIIをマスクしてクラウドAIへ送信 | ✅ **推奨** |
| **D: Cloud Only** | 生データをクラウドAIへ送信（マスクなし） | — |

現時点では **Mode C（Masked Cloud）** が唯一の実用的な選択肢です。「クラウドに送るのは不安だが、ローカルAIはまだ不安定」というニーズに対して、マスキングによる折衷案として機能しています。

## プライベートページ検出 — HTTPヘッダーを読む

「自動保存時に確認する」設定が効くのは、サーバーが返す **HTTPレスポンスヘッダー** に特定のプライバシー信号が含まれている場合だけです。

### 検出するヘッダー

| ヘッダー | 意味 | 典型的なサイト |
|---------|------|--------------|
| `Cache-Control: private` | このレスポンスはプライベートなキャッシュにのみ保存してよい | ネットバンキング・EC注文履歴 |
| `Set-Cookie` | セッション管理Cookieが含まれる（ログイン後の画面など） | 社内ポータル・SNS |
| `Authorization` | 認証が必要なコンテンツ | REST API・社内システム |

警視庁の公開ページはこれらのヘッダーを返しません。公開情報なのでCDNキャッシュも許可されています。そのためプライベートページ検出は発動せず、確認通知も出なかったんです。

### なぜ `no-cache` は対象外なのか

「`Cache-Control: no-cache` は対象にならないのか？」という疑問が出るかもしれません。

実は、国税庁・警視庁など多くの公官庁サイトが `no-cache` を返しています。これを含めてしまうと、普通の公開ページが大量にスキップされてしまいます。誤検出が多すぎて使い物にならなくなるため、意図的に対象から外しました。

---

## どんな場面で確認通知が出るか

確認通知（Chrome通知の「保存しますか？」）が出るのは、サーバー側がプライベートと明示しているページを訪れたときです。

![スキップされたページがダッシュボードに残る様子](image-4.png)

典型的な例をあげると、

- ネットバンキングのマイページ（ログイン後）
- 社内ポータルや業務システム
- 医療機関の患者向けポータル
- ECサイトの注文履歴・アカウントページ

これらのページではサーバーが `Cache-Control: private` や `Set-Cookie` ヘッダーを返すことが多いです。ブラザーのサポートページ（`https://faq.brother.co.jp/...`）のように、一見して公開情報に見えても `Cache-Control: private, max-age=180` を返すケースもあります。

## よくある誤解

### Q. 銀行サイトでも「🔒マスクあり」が出た。確認通知は出なかったのに大丈夫？

大丈夫です。順番に整理しますね。

1. ページ読み込み時にHTTPヘッダーを確認 → `Cache-Control: private` を検出
2. 設定が `confirm` なら Chrome通知が出る（`save` ならそのまま保存）
3. 保存後、ページのテキストにPIIが含まれていればAI送信前にマスク → 🔒バッジ

「確認通知が出なかった」なら、設定が `save`（デフォルト）になっている可能性が高いです。プライベートページ検出は発動していたけれど、設定通り確認なしで保存した、ということです。

設定を変えたい場合は、ダッシュボード → Privacy タブ → Confirmation Settings で `confirm` を選んでください。

### Q. `confirm` 設定にしているのに通知が出ないページがある。バグ？

バグではありません。HTTPヘッダーでプライベートと宣言していないページは検出対象外です。公開されているページにはプライベートページ検出は発動しません。

テキスト内のPIIはPIIマスクが別途保護しています。どちらの保護も動いていない場合は、そもそもPIIが含まれていないか、ヘッダーがプライベートを示していないページだということです。

### Q. 両方が同時に発動するケースはある？

あります。ネットバンキングのマイページが典型例です。

- `Cache-Control: private` ヘッダーあり → プライベートページ検出が発動
- ページ内に口座番号・メールアドレスが記載 → PIIマスクも発動

この場合、確認通知（プライベートページ検出）が出て、保存を選択した後、PIIをマスクしてからAIに送信されます（🔒バッジが記録に残ります）。

## まとめ

警視庁のページが「🔒マスクあり」で保存されたのは正常な動作でした。

- **PIIマスク** — テキスト内の電話番号等を検出してAI送信前にマスク → 正しく動作した
- **プライベートページ検出** — HTTPヘッダーを確認 → 公開ページなので対象外

「確認設定」で守れるのはHTTPヘッダーでプライベートと宣言されたページだけです。テキスト内の個人情報は、設定によらずPIIマスクが自動的に保護します。

それぞれの役割を理解しておくと、ダッシュボードのバッジから「どのプライバシー保護が働いたか」を読み取れるようになります。

---

## 補足: スキップされたページはどこに残るか

「保存をスキップ」設定にしている場合、プライベートページ検出が発動したページはObsidianには書き込まれません。ではそのページはどこへ行くのか。

ダッシュボードの記録履歴に答えがあります。

![記録できなかったページがダッシュボードに残る様子](image-4.png)

スキップされたページは「Skipped」フィルターに保持され、次の情報が表示されます。

- ページタイトルとURL
- スキップされた日時
- 検出されたヘッダーの値（オレンジ色で表示）

「今すぐ記録」ボタンを押せばその場で手動保存できます。自動記録はしたくないけれど、内容を確認してから判断したい、というケースに対応しています。

スキップされたページの保持期間は24時間です。それを過ぎると自動的に削除されます。
