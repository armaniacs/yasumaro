## 1. マスターパスワードレート制限実装

### 機能概要
マスターパスワード認証に対してレート制限を実装し、ブルートフォース攻撃を防止します。

### 入出力仕様

**入力**:
- パスワード文字列 (string)
- 認証試行履歴 (chrome.storage.session)

**出力**:
- 認証成功: `{ success: true }`
- 認証失敗: `{ success: false, error: string }`
- レート制限超過: `{ success: false, error: 'Too many attempts. Please try again in X minutes.' }`

### 制約条件
- 5分以内の認証失敗が5回を超えると、30分間ロックアウト
- ロックアウト開始時刻は `chrome.storage.session` に保存
- ロックアウト終了後、失敗回数カウンターはリセット

### 🟢🟡🔴 信頼性レベル
このセクションの信頼性レベル: 🟢 **青信号**（要件定義書および既存コードに基づいて推測が最小限）

---

- **期待される動作**: 認証成功時に失敗回数カウンターは0のまま
- 🟢 **信頼性レベル**: 青信号（既存コードのverifyMasterPassword動作から推測）

**テスト名**: 認証失敗時、失敗回数カウンターが増加する

- **期待される動作**: 認証失敗時に失敗回数が1増加
- 🟢 **信頼性レベル**: 青信号（要件定義書の動作仕様通り）

**テスト名**: 5回認証失敗後、ロックアウト状態になる
- **何をテストするか**: 5回連続で認証に失敗した場合、ロックアウトが発動することを確認
- **期待される動作**: 6回目の認証試行時、ロックアウトエラーが返される

**テスト名**: 30分後、ロックアウトが解除される

- 🟢 **信頼性レベル**: 青信号（要件定義書の仕様通り）

**テスト名**: 認証成功後、失敗回数カウンターがリセットされる

## 2. コンソールログ機密情報削除

### 機能概要
コンソールログに記録される機密情報（APIキー等）をマスクまたは削除します。

### 入出力仕様

**入力**:
- ログ出力するデータ (any型)

**出力**:
- 機密情報がマスクされたログ文字列 (string)

### 制約条件
- APIキー、パスワード、トークンなどの機密文字列は `***MASKED***` またはログ出力から排除
- デバッグ情報としての型情報や長さは保持可能

### 🟢🟡🔴 信頼性レベル
このセクションの信頼性レベル: 🟢 **青信号**（既存コードの問題箇所が特定済み）

---

#### 正常系テストケース

**テスト名**: APIキーがログ出力から除外される
- **何をテストするか**: console.errorにAPIキーを含むオブジェクトを渡した場合、ログにAPIキーが含まれないことを確認
- **期待される動作**: ログ出力に `fullKey: 'sk-...'` が含まれていない
- 🟢 **信頼性レベル**: 青信号（既存コード obsidianClient.ts:120 の問題箇所確認済み）

**テスト名**: APIキーの型情報のみログに出力される
- **何をテストするか**: APIキーの型情報 (`typeof`) はログに出力可能であることを確認
- **期待される動作**: ログ出力に `apiKey: 'string'` が含まれる
- 🟢 **信頼性レベル**: 青信号（既存コードの動作仕様通り）

**テスト名**: addLog 関数が正しく機密情報を処理する
- **何をテストするか**: logger.tsのaddLog関数がdetailsに機密情報が含まれる場合、適切に処理することを確認
- **期待される動作**: detailsオブジェクトから機密情報が除外またはマスクされる
- 🟢 **信頼性レベル**: 青信号（logger.ts のドキュメントに「NO RAW PII」と記載あり）

---

#### 異常系テストケース

**テスト名**: 機密情報を含む深いネストオブジェクトの処理
- **エラーケースの概要**: 深くネストされたオブジェクトにAPIキーが含まれる場合
- **エラー処理の重要性**: 任意の深さのオブジェクト構造でも機密情報を保護
- 🟡 **信頼性レベル**: 黄信号（apiキーの標準的なキー名について推測）

**テスト名**: 配列内の機密情報処理
- **エラーケースの概要**: 配列形式のデータにAPIキーが含まれる場合
- **エラー処理の重要性**: 配列要素内の機密情報も保護
- 🟡 **信頼性レベル**: 黄信号（実装方針による）

---

#### 境界値テストケース

**テスト名**: 空文字列のAPIキー処理
- **境界値の意味**: 空のAPIキー値に対するログ動作確認
- **境界値での動作保証**: 偽陽性を回避
- 🟢 **信頼性レベル**: 青信号（既存コードのバリデーションロジックから推測）

**テスト名**: null/undefined APIキー処理

- 🟢 **信頼性レベル**: 青信号（既存コードのバリデーションロジックから推測）

## 3. デバッグログ本番無効化

### 機能概要
本番環境ではDEBUGレベルのログを出力しないように、環境に応じたログレベルフィルタリングを実装します。

### 入出力仕様

**入力**:
- ログタイプ (LogTypeValues)
- ログメッセージ (string)
- 詳細情報 (Record<string, any>)

**出力**:
- ログストレージへの書き込み (環境依存)

### 制約条件
- 本番環境: INFO, WARN, ERROR, SANITIZE のみ出力
- 開発環境: DEBUG も含めて出力
- 環境判定: process.env.NODE_ENV または拡張のマニフェスト設定

### 🟢🟡🔴 信頼性レベル
このセクションの信頼性レベル: 🟡 **黄信号**（環境判定方法について推測が必要）

---

#### 正常系テストケース

**テスト名**: 本番環境のDEBUGログが出力されない
- **何をテストするか**: NODE_ENV=production 状態でDEBUGログを追加した場合、storageに保存されないことを確認
- **期待される動作**: DEBUGレベルのログは破棄される
- 🟡 **信頼性レベル**: 黄信号（環境判定方法が確定していない）

**テスト名**: 本番環境のERRORログが出力される
- **何をテストするか**: NODE_ENV=production 状態でERRORログを追加した場合、正しく保存されることを確認
- **期待される動作**: ERRORレベルのログはstorageに保存される
- 🟡 **信頼性レベル**: 黄信号（環境判定方法が確定していない）

**テスト名**: 開発環境のDEBUGログが出力される
- **何をテストするか**: NODE_ENV=development 状態でDEBUGログを追加した場合、正しく保存されることを確認
- **期待される動作**: DEBUGレベルのログはstorageに保存される
- 🟡 **信頼性レベル**: 黄信号（環境判定方法が確定していない）

---

#### 異常系テストケース

**テスト名**: 未定義のノード環境でのデフォルト挙動
- **エラーケースの概要**: NODE_ENVが設定されていない場合
- **エラー処理の重要性**: 安全デフォルトを適用するか確認
- 🔴 **信頼性レベル**: 赤信号（要件定義書に記載なし、実装方針による）

**テスト名**: 不正な環境文字列の処理
- **エラーケースの概要**: NODE_ENVに無効な値が設定されている場合
- **エラー処理の重要性**: 実装ミスによるログ流出防止
- 🔴 **信頼性レベル**: 赤信号（要件定義書に記載なし）

---

#### 境界値テストケース

**テスト名**: ログ型列挙値の全種類が正しく扱われる
- **境界値の意味**: LogTypeの全ての値（INFO, WARN, ERROR, SANITIZE, DEBUG）が正しく処理されるか確認
- **境界値での動作保証**: 将来のログタイプ追加時の安全確認

**テスト名**: 空メッセージのログ追加
- **境界値の意味**: 空文字列のログメッセージを追加した場合
- **境界値での動作保証**: エラー発生せず正常処理
- 🟡 **信頼性レベル**: 黄信号（実装の挙動による）

**テスト名**: 最大サイズの詳細情報オブジェクト
- **境界値の意味**: 非常に大きなdetailsオブジェクトをログ追加した場合

## 4. 設定ファイル署名強化

### 機能概要
署名のない設定ファイルのインポートを拒否し、署名検証のセキュリティを強化します。

### 入出力仕様

**入力**:
- 設定ファイルのJSON文字列 (string)

**出力**:
- インポート成功: Settings オブジェクト
- インポート失敗: null

### 制約条件
- 署名のないファイルは一律拒否
- 署名検証に失敗したファイルは一律拒否
- HMAC-SHA256を使用した署名検証

### 🟢🟡🔴 信頼性レベル
このセクションの信頼性レベル: 🟢 **青信号**（既存コードの問題箇所が特定済み）

---

#### 正常系テストケース

**テスト名**: 有効な署名付き設定ファイルのインポート成功
- **何をテストするか**: 正しい署名が付いた設定ファイルをインポートした場合、成功することを確認
- **期待される動作**: Settingsオブジェクトが返される
- 🟢 **信頼性レベル**: 青信号（既存コード settingsExportImport.ts:367-381 の署名検証ロジック確認済み）

**テスト名**: HMAC署名の正確性検証
- **何をテストするか**: エクスポート時の署名と計算した署名が一致することを確認
- **期待される動作**: 同じデータに対して署名が一致する
- 🟢 **信頼性レベル**: 青信号（settingsExportImport.ts:374 の署名計算ロジック確認済み）

**テスト名**: 署名が改ざんされたファイルのインポート失敗
- **何をテストするか**: 署名が変更された設定ファイルをインポートした場合、失敗することを確認
- **期待される動作**: nullが返され、アラートが表示される
- 🟢 **信頼性レベル**: 青信号（既存コード settingsExportImport.ts:376-380 のエラーハンドリング確認済み）

**テスト名**: データが改ざんされたファイルの署名検証失敗
- **何をテストするか**: 設定データ内の値を変更した場合、署名検証が失敗することを確認
- **期待される動作**: nullが返される
- 🟢 **信頼性レベル**: 青信号（署名の整合性特性から推測）

- **エラーケースの概要**: signatureフィールドがnullまたは空文字列の場合
- **エラー処理の重要性**: 実装ミスによる署名バイパス防止
- 🟢 **信頼性レベル**: 青信号（352行の `!parsed.signature` チェックで対応）

#### 境界値テストケース

**テスト名**: 空のsettingsオブジェクトの署名検証
- **境界値の意味**: 空の設定オブジェクトに対する署名動作確認

## 品質判定

✅ **高品質**

### 判定理由

1. **テストケース分類**: 正常系・異常系・境界値が網羅されている
   - 正常系: 15件（各機能の基本動作を網羅）
   - 異常系: 8件（セキュリティ上の重要なエラーケースを含む）
   - 境界値: 12件（システム安定性確認のための境界条件を網羅）

2. **期待値定義**: 各テストケースの期待値が明確
   - 失敗回数の増減
   - ログ出力の有無
   - 戻り値の型と値

3. **技術選択**: プログラミング言語・テストフレームワークが確定
   - TypeScript 5.9.3
   - Jest 29.7.0 + jsdom
   - @peculiar/webcrypto 1.5.0

4. **実装可能性**: 現在の技術スタックで実現可能
   - 既存のchrome.storage.local mockを拡張してsession mockを追加
